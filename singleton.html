Singleton Pattern

Creational Design Pattern — F1 Race Director Control Centre

01

What is the Singleton Pattern?

A creational design pattern that ensures a class has only one instance and provides a global point of access to it — like a single Race Director who controls the entire Grand Prix.

📊 UML Structure

┌──────────────────────────────────────┐ │ Singleton │ ├──────────────────────────────────────┤ │ - static _instance: Singleton │ │ - state: object │ ├──────────────────────────────────────┤ │ + static getInstance(): Singleton│ │ + getState() │ │ + setState(data) │ └──────────────────────────────────────┘ Client A ───┐ ├──▶ getInstance() ─▶ same object Client B ───┘

🏁 F1 Race Director Analogy

In Formula 1, there is exactly one Race Director per Grand Prix. Every team, marshal, and broadcast refers to the same person. If two directors existed, conflicting flags and penalties would cause chaos.

  • 🔴 Race Director — decides flags, safety cars, penalties
  • 📡 Timing System — one master clock for all teams
  • 🎙️ Radio Channel — single broadcast frequency for race control
  • ☁️ Weather Station — one official feed for the entire circuit

Each of these is a real-world Singleton — only one instance makes sense, and everyone shares it.

📚 Key Concepts

  • Singleton — the class itself, which stores its sole instance in a static field
  • getInstance() — the public static method that returns the one instance (creating it on first call)
  • Private constructor — prevents external code from using new
  • Lazy initialisation — the instance is created only when first requested
  • Global access point — any part of the application can reach the singleton

💻 Core Implementation

// Singleton: only one Race Director exists
class RaceDirector {
  static _instance = null;

  constructor() {
    if (RaceDirector._instance)
      return RaceDirector._instance;
    this.flags = [];
    RaceDirector._instance = this;
  }

  static getInstance() {
    if (!RaceDirector._instance)
      new RaceDirector();
    return RaceDirector._instance;
  }
}

// Both variables reference the SAME object
const a = RaceDirector.getInstance();
const b = RaceDirector.getInstance();
a === b; // → true

How the Pattern Works Step by Step

The singleton guarantees a single shared instance throughout the application lifecycle.

1⃣ First Request

Example: The Team Principal calls RaceDirector.getInstance() at the start of the weekend. No instance exists yet, so the constructor creates one, stores it in the static field, and returns it.

2⃣ Subsequent Requests

Example: A Marshal calls RaceDirector.getInstance() during the race. The static field already holds the instance, so the exact same object is returned — no new object created.

3⃣ Shared State

Example: The Principal sets a Yellow Flag. When Broadcast reads the race status, it sees the yellow flag — because both are referencing the same singleton object and its state.

4⃣ Identity Guarantee

Example: principalRef === marshalRef evaluates to true. JavaScript’s strict equality proves both variables point to the identical object in memory.

04

Benefits of the Singleton Pattern

When and why to use a singleton in your architecture.

🔐 Controlled Access

Example: Just as only one Race Director can deploy a safety car, the singleton prevents rogue instances from corrupting shared state like databases, caches, or config objects.

💾 Reduced Memory

Example: Instead of each F1 team creating its own weather station, they share one. Similarly, singletons avoid duplicating heavy resources like connection pools or thread pools.

🛠️ Global Coordination

Example: All marshals see the same flag status because they share one director. In code, loggers, event buses, and app-wide settings benefit from singleton coordination.

🌎 Real-World Usage

Examples: Java’s Runtime.getRuntime(), Python’s logging module, Angular’s root-scoped services, database connection pools, browser window object.

05

Singleton vs Global Variable

Why a controlled singleton beats a plain global.

Criteria Global Variable Singleton Pattern 🔐
Instance control None — anyone can overwrite Strict — exactly one instance
Lazy initialisation Loaded immediately at startup Created only when first requested
Encapsulation State fully exposed State behind methods
Testing Hard to isolate Can reset/mock the instance
Thread safety No guarantees Can lock during creation
