Adaptive Pattern

Structural Design Pattern — Unifying F1 telemetry feeds

01

What is the Adaptive Pattern?

This demo uses the term Adaptive for a pattern that adapts incompatible interfaces into one client-friendly API. In classic GoF terminology, this is the Adapter pattern.

F1 telemetry analogy

A race engineer needs one dashboard, but telemetry arrives from different systems with different field names and formats.

  • Ferrari Garage Unit reports raw milliseconds, tyre codes and ERS percentage
  • FIA Timing Feed sends lap time as a string plus fuel in litres
  • Weather Radar focuses on track grip, wind and rain probability

The client should not care about those differences. It asks every source for one normalized snapshot.

Key concepts

  • Client depends on one stable interface
  • Adaptee is an existing class with an incompatible API
  • Adapter translates calls and data shapes
  • The adapter can also convert units, defaults and naming conventions

Core implementation

class FiaTimingAdapter {
  constructor(feed) {
    this.feed = feed;
  }

  getSnapshot() {
    const raw = this.feed.read();
    return {
      source: 'FIA Timing Feed',
      driver: raw.driverNumber,
      lapTime: raw.lap_time,
      tyre: raw.stint,
      energy: raw.battery,
      fuel: `${raw.fuel_remaining_l} L`
    };
  }
}

03

How the adaptation flow works

The adapter isolates legacy complexity and gives client code a stable contract.

1. Legacy provider keeps its own API

No existing telemetry system has to be rewritten. Each source keeps its original method names and payload shape.

2. Adapter translates shape and units

The adapter converts milliseconds to formatted lap times, tyre codes to human-readable values and different field names into one contract.

3. Client depends on one interface

The dashboard always calls getSnapshot(), so it stays decoupled from Ferrari, FIA or weather-specific details.

4. New feeds are cheap to add

When a new partner system appears, you add one more adapter instead of changing the client or rewriting the legacy service.
