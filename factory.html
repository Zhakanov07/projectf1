Factory Pattern

Creational Design Pattern — F1 Component Manufacturing

01

What is the Factory Pattern?

A creational design pattern that provides an interface for creating objects in a superclass, but allows subclasses to alter the type of objects that will be created — without exposing instantiation logic to the client.

F1 Manufacturing Analogy

Each F1 team has its own factory that produces team-specific components. The FIA regulates the interface — every component must meet certain specs — but each factory builds them differently:

  • Ferrari Factory — produces high-power Italian-engineered parts
  • Red Bull Factory — produces aero-focused components by Adrian Newey
  • McLaren Factory — produces balanced, reliable British components
  • Mercedes Factory — produces precision German-engineered parts

Key Concepts

  • Product — defines the interface for objects the factory method creates
  • ConcreteProduct — specific implementation of the product (e.g. Ferrari Engine, Red Bull Chassis)
  • Creator — declares the factory method that returns a Product object
  • ConcreteCreator — overrides the factory method to produce a specific ConcreteProduct
  • The client code works with factories and products via interfaces, staying decoupled from concrete classes

Core Implementation

class TeamFactory {
  createEngine()  { throw 'Override me!'; }
  createChassis() { throw 'Override me!'; }
}

class FerrariFactory extends TeamFactory {
  createEngine() {
    return new FerrariEngine();
  }
  createChassis() {
    return new FerrariChassis();
  }
}

// Client code works with the factory interface
const factory = new FerrariFactory();
const engine  = factory.createEngine();
engine.getSpecs();  // → Ferrari V6 Turbo Hybrid

03

How the Pattern Works Step by Step

The factory encapsulates object creation, letting subclasses decide which class to instantiate.

1⃣ Define the Product Interface

Create a common interface (e.g. getSpecs(), getPower()) that all products must implement. This guarantees the client can work with any product uniformly.

2⃣ Implement Concrete Products

Each team creates its own version: FerrariEngine outputs 1000 HP, RedBullEngine outputs 1020 HP. Different internals, same interface.

3⃣ Create Factory Classes

FerrariFactory overrides createEngine() to return a FerrariEngine. The client never calls new FerrariEngine() directly.

4⃣ Client Uses Factory Interface

The client calls factory.createEngine(). It gets back an engine without knowing (or caring) which concrete class was instantiated. Swap factories to change products.

04

Benefits of the Factory Pattern

Why this pattern is essential for scalable object creation.

🔒 Loose Coupling

Client code depends only on the abstract Creator and Product interfaces, not on concrete classes. You can add new teams without modifying existing code.

🛠️ Single Responsibility

Object creation logic is centralized in factory classes. The client focuses on using products, not constructing them.

🔄 Easy Substitution

Swap FerrariFactory for RedBullFactory and the entire product family changes. Ideal for configuration-driven or plugin-based architectures.

🌎 Real-World Usage

Java’s Calendar.getInstance(), React’s createElement(), Python’s logging.getLogger(), database driver factories, UI theme factories.

05

Factory vs Direct Instantiation

Why delegating creation beats hard-coded constructors.

Criteria Direct new Factory Pattern 🏭
Coupling Tight — client knows concrete classes Loose — client knows interfaces only
Adding new types Modify client code everywhere Add a new factory subclass
Testing Hard to mock concrete constructors Inject mock factory easily
Flexibility Fixed at coding time Selectable at runtime
Example (4 teams, 4 parts) 16 scattered new calls 4 factory classes, 1 interface call
