Decorator Pattern

Structural Design Pattern — F1 Car Upgrade Builder

01

What is the Decorator Pattern?

A structural design pattern that lets you dynamically add new behaviours to objects by wrapping them in special decorator objects, without altering the original class.

F1 Upgrade Analogy

In our demo, a base F1 car is the Component. Each upgrade is a Decorator that wraps the car and modifies its specs:

  • DRS System — +15 km/h top speed
  • Hybrid Boost — +50 HP, +12 kg
  • Lightweight Chassis — −30 kg
  • Aero Package — +8 km/h downforce
  • Slick Tyres — +5 km/h grip
  • KERS Recovery — +30 HP, +8 kg

Key Concepts

  • Component — defines the interface for objects that can have responsibilities added dynamically
  • ConcreteComponent — the original object being decorated (e.g. base F1 car)
  • Decorator — abstract wrapper that holds a reference to a Component and delegates calls
  • ConcreteDecorator — adds specific behaviour (e.g. DRS, Hybrid Boost, Aero Package)
  • Decorators are stackable — multiple can wrap the same object

Core Implementation

class BaseCar {
  getTopSpeed()   { return 0; }
  getHorsepower() { return 0; }
}

class FerrariSF23 extends BaseCar {
  getTopSpeed()   { return 340; }
  getHorsepower() { return 1000; }
}

// Base Decorator wraps a BaseCar
class CarDecorator extends BaseCar {
  constructor(car) { this._car = car; }
  getTopSpeed()   { return this._car.getTopSpeed(); }
}

// Concrete Decorator
class DRSUpgrade extends CarDecorator {
  getTopSpeed() {
    return this._car.getTopSpeed() + 15;
  }
}

// Usage: stack decorators
let car = new FerrariSF23();
car = new DRSUpgrade(car);
car.getTopSpeed(); // → 355

How the Pattern Works Step by Step

The decorator chain processes method calls through all wrappers.

1⃣ Create Base Component

You start with a FerrariSF23 — a concrete component with base stats: 340 km/h, 1000 HP, 798 kg.

2⃣ Wrap with Decorator

new DRSUpgrade(car) creates a decorator that holds a reference to the original car and overrides getTopSpeed() to add +15.

3⃣ Stack More Decorators

new HybridBoost(car) wraps the already-decorated car. Each decorator delegates non-overridden methods to the inner object.

4⃣ Transparent to Client

The client calls car.getTopSpeed() on the outermost wrapper. The call propagates through the chain, each decorator adding its modification.

04

Benefits of the Decorator Pattern

Why this pattern is powerful for extensibility.

Single Responsibility

Each decorator has exactly one responsibility: add DRS, add Hybrid, etc. The base car class stays clean and unchanged.

Open/Closed Principle

New upgrades can be added as new decorator classes without modifying existing car or decorator code.

Runtime Composition

Unlike inheritance (which is static), decorators can be added and combined at runtime in any order, creating flexible configurations.

Real-World Usage

Java I/O streams (BufferedReader wraps FileReader), middleware in Express.js, Python's @decorator syntax, React Higher-Order Components.

05

Decorator vs Inheritance

Why wrapping beats subclassing for flexible extensions.

Criteria Inheritance Decorator Pattern
Flexibility Static — fixed at compile time Dynamic — compose at runtime
Combinations Exponential class explosion Any combination of decorators
Adding features New subclass for each combo New decorator class only
Removing features Impossible without new class Simply unwrap decorator
Example (6 upgrades) 64 subclasses (2⁶) 6 decorator classes
