Command Pattern

Behavioural Design Pattern — F1 Pit Wall Strategy Console

01

What is the Command Pattern?

A behavioural design pattern that encapsulates a request as an object, letting you parameterize clients, queue requests, log them, and support undoable operations.

🏁 F1 Pit Wall Analogy

During a race, the Pit Wall (invoker) sends strategic commands to the car. Each command is a distinct object that can be executed, queued, or undone:

  • 🔄 “Box Box!” — pit stop command, can undo by aborting the stop
  • ⚡ “Deploy ERS” — energy boost command, reversible by harvesting
  • 💨 “DRS Open” — drag reduction command, undo by closing DRS
  • 🌱 “Switch to Soft” — tyre strategy command, undo reverts to previous compound
  • 🚀 “Push Mode” — engine mode command, undo returns to normal
  • 🔧 “Adjust Front Wing” — aero command, stores previous angle for undo

The driver (receiver) executes the command, and the pit wall can undo any decision by reverting to the previous state — crucial in fast-paced racing.

📚 Key Concepts

  • Command — interface declaring execute() and undo()
  • ConcreteCommand — implements the action (e.g. PitStopCommand, DRSCommand)
  • Receiver — the object that performs the actual work (the F1 car)
  • Invoker — triggers commands without knowing their internals (the Pit Wall)
  • Command history — stack of executed commands enabling undo/redo
  • Decoupling — invoker doesn’t know which receiver or how work is done

💻 Core Implementation

// Command interface
class Command {
  execute() { throw 'Override!'; }
  undo()    { throw 'Override!'; }
}

// Concrete command with undo
class DRSCommand extends Command {
  constructor(car) { this.car = car; }
  execute() { this.car.drs = true;  }
  undo()    { this.car.drs = false; }
}

// Invoker keeps a history stack
class PitWall {
  history = [];
  executeCmd(cmd) {
    cmd.execute();
    this.history.push(cmd);
  }
  undoLast() {
    this.history.pop()?.undo();
  }
}

03

How the Pattern Works Step by Step

Commands encapsulate actions and enable undo by storing previous state.

1⃣ Create a Command

Example: The engineer creates a PitStopCommand(car). The command holds a reference to the car (receiver) and stores its current tyre compound for potential undo.

2⃣ Invoker Executes It

Example: The Pit Wall calls pitWall.executeCmd(pitStopCmd). The invoker calls cmd.execute() and pushes it to the history stack, without knowing what the command does internally.

3⃣ Undo Via History

Example: Rain starts unexpectedly! The Pit Wall calls pitWall.undoLast(), which pops the last command and calls cmd.undo() to revert the car to its previous state.

4⃣ Queue & Replay

Example: Commands can be queued: “After lap 20, deploy ERS, then open DRS.” They can also be serialized, logged, or replayed for telemetry analysis.

04

Benefits of the Command Pattern

Why encapsulating actions as objects is so powerful.

⏪ Undo / Redo

Example: In an F1 strategy sim, every decision (pit, tyres, engine mode) can be undone. Just like Ctrl+Z in your editor — each keystroke is a command object with an inverse operation.

📋 Logging & Audit

Example: Every pit call is logged as a command object with timestamp, parameters, and result. Race analysts replay the entire strategy by iterating the command history.

🔄 Decoupled Invoker

Example: The pit wall button “Box Box” doesn’t know if it triggers a tyre change, fuel adjustment, or wing tweak — it just calls execute() on whatever command is assigned.

🌎 Real-World Usage

Examples: Text editor undo stacks, database transaction rollback, game input replay, React Redux actions, macOS NSUndoManager, task scheduling queues.

05

Command vs Direct Method Call

Why wrapping actions in objects beats calling methods directly.

Criteria Direct Call Command Pattern 📋
Undo support None — must manually track state Built-in via undo() method
Logging Ad-hoc, scattered Automatic — commands are objects
Queueing Not possible Commands can be queued and deferred
Coupling Caller knows the receiver Invoker is decoupled from receiver
Replay / macro Impossible without recording Replay history array
