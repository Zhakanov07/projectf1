Observer Pattern

Behavioral Design Pattern — F1 Race Control Demo

01

What is the Observer Pattern?

A behavioural design pattern that defines a one-to-many dependency between objects so that when one object (Subject) changes state, all its dependents (Observers) are notified automatically.

UML Structure

┌──────────────────────────┐ │ «Subject» │ │ EventSubject │ ├──────────────────────────┤ │ - observers: Map │ ├──────────────────────────┤ │ + subscribe(type, obs) │ │ + unsubscribe(type, obs) │ │ + notify(type, data) │ └──────────┬───────────────┘ │ extends ┌──────────▼───────────────┐ │ RaceControl │ ├──────────────────────────┤ │ + setFlag(flag) │ │ + setWeather(weather) │ │ + deploySafetyCar(bool) │ │ + sendMessage(text) │ └──────────────────────────┘ │ notifies ──▶ ┌────────────────┐ │ │ «Observer» │ ├──────────────▶│ TeamRadio │ ├──────────────▶│ PitWall │ ├──────────────▶│ TVBroadcast │ └──────────────▶│ MarshalPost │ ├────────────────┤ │ + update(t, d) │ └────────────────┘

F1 Race Analogy

In our demo, Race Control is the Subject. It emits events like flag changes, weather updates, and safety car deployments. The observers are:

  • Team Radio — receives flags, safety & messages
  • Pit Wall — receives ALL events (full strategy)
  • TV Broadcast — receives flags, weather & safety
  • Marshal Post — receives flag events only

Key Concepts

  • Subject — maintains a list of observers and notifies them of state changes
  • Observer — defines an update() interface that gets called by the Subject
  • subscribe() — registers an observer for specific events
  • unsubscribe() — removes an observer from the notification list
  • notify() — iterates over observers and calls their update method

Core Implementation

class EventSubject {
  constructor(name) {
    this._observers = new Map();
  }

  subscribe(eventType, observer) {
    if (!this._observers.has(eventType))
      this._observers.set(eventType, new Set());
    this._observers.get(eventType).add(observer);
  }

  notify(eventType, data) {
    for (const obs of this._observers.get(eventType))
      obs.update(eventType, data);
  }
}

// Concrete Subject
class RaceControl extends EventSubject {
  setFlag(flag) {
    this.notify('flag', { flag });
  }
}

// Observer
class TeamRadio {
  update(eventType, data) {
    console.log(`Radio: ${eventType}`, data);
  }
}

How the Pattern Works in This Demo

Step-by-step flow of an event through the Observer pattern.

1⃣ Subject Emits Event

Race Control calls setFlag('yellow'), which internally calls notify('flag', { flag: 'yellow', lap: 12 }).

2⃣ Observer Map is Checked

The Subject looks up 'flag' in its _observers Map and finds all subscribed observers.

3⃣ Each Observer is Notified

The Subject iterates over the Set and calls observer.update('flag', data) on each one.

4⃣ Observers React Independently

Each observer processes the event in its own way — Team Radio gives advice, Pit Wall recalculates strategy, TV shows info to viewers, Marshals wave flags.

04

Benefits of the Observer Pattern

Why this pattern is used in real-world software and our F1 project.

Loose Coupling

The Subject doesn't need to know the concrete classes of its observers. It only relies on the update() interface, making the system flexible and modular.

Open/Closed Principle

New observers can be added without modifying the Subject class. We can add a DataLogger or SocialMedia observer without touching Race Control.

Dynamic Subscriptions

Observers can subscribe and unsubscribe at runtime. Toggle checkboxes in the demo above to see this in action!

Real-World Usage

Event listeners in DOM (addEventListener), RxJS observables, Vue.js reactivity, Redux store subscriptions, WebSocket message handlers — all use the Observer pattern.
